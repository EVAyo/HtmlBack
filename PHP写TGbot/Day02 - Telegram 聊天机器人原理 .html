今天要来讲讲机器人是怎么运作的

Client-Server 架构

这是一种网路架构，网页大部分都是使用此架构
瀏览器是客户端(client)，客户端(client)主动向伺服器(server)发送请求(request)
伺服器(server)会执行一些程式，之后把页面被动回应(response)给客户端(client)

伺服器(server)需要一个 IP 位置，这样子瀏览器才知道要把请求(request)发到哪个地方
IP 位置就像是住家的地址一样，而请求(request)就像是一封信

机器人的运作原理


现在你发了一段讯息给机器人
但是並不是机器人直接回你讯息

详细情况如下图：

Icon 来源：https://www.flaticon.com/packs/programming-29

  1. 一开始你的讯息会被传送到 TG 伺服器
  2. TG 伺服器会通知机器人：欸！有一则新讯息
  3. 机器人会发一个指令给 TG 伺服器
  4. TG 伺服器照著指令的需求，回应讯息给你

机器人接收讯息的两种模式

机器人可以放在伺服器，也可以用客户端的方式运作

  1. Webhook
    以伺服器方式被动接收讯息，需要有 IP 位置，通常会放在网站伺服器(Web Hosting)上，要自己架设网站伺服器不是件容易的事，所以官方还提供了另一种做法
  2. Long Polling
    以客户端方式主动跟 TG 伺服器要讯息，可以直接在自己电脑上运作，不需要架(租)伺服器也能让机器人顺利运作，缺点就是电脑要一直开著，当然你也可以使用云端服务(IaaS、PaaS、SaaS)

机器人发送讯息的三种方式

  1. HTTP GET

    https://api.telegram.org/bot{Token}/{Method}?{Parameter1}={Value1}&{Parameter2}={Value2}

    直接以网址参数的方式传送指令，GET 没有 Body，只传送 Headers (你可以当作不会传送档案过去)，通常是用来取资料用，优点是可以直接用瀏览器执行，缺点是有 2KB 的限制，虽然不太可能用到这长的指令，但还是比较建议使用 POST

  2. HTTP POST

    {"method":"Method","Parameter1":"Value1","Parameter2":"Value2"}

    POST 是以 Body 传送资料(档案)，程式通常都是以这种方式沟通，网页里的表单也经常是以这种方式传送，资料不会显示在网址列，可以传送的资料大小比 GET 大得多：1MB ~ 2GB

  3. 直接 Print

    这个方式只有在 webhook 模式下才能使用，因为只有伺服器可以直接回应讯息
